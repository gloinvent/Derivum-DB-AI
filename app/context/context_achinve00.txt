## 1. DATABASE ARCHITECTURE

Two logical databases:
- **PDB (Primary Database):** Bond master data — ISINs, issuers, ratings, redemptions, cashflows, covenants, EBP issuance records.
- **SDB (Secondary Database):** Secondary market trade data — individual trades, daily averages, 15-day rolling averages.

**Central table:** `PDB_isin_records` (aliased `ir`) is the hub. Most queries start FROM or JOIN through this table.

**Join philosophy (CRITICAL — follow Praful's pattern):**
- Primary join path: `ir.issuer_organization_id = io.id` (ID-based, never text-match on issuer_name).
- EBP join: `e.isin_id = ir.id` — use only when you need EBP-specific columns (yield_ebp, spread_bps, total_issue_size, bidding_date, allotment columns). Do NOT join PDB_ebp_records for ISIN-level attributes that exist on PDB_isin_records (face_value, coupon_fixed, seniority, etc.).
- Trade join: `t.isin_record_id = ir.id` for SDB_trade.
- 15-day avg join: `f.isin = ir.isin` (text match on ISIN code) for SDB_fifteen_days_trade_avg.
- All child tables (PDB_redemption, PDB_payin, PDB_tag, PDB_current_rating_agency, PDB_cashflow_record, PDB_isin_security, PDB_investor, PDB_im_records, PDB_call_option_dates, PDB_put_option_dates): join via `child.isin_id = ir.id` (except PDB_cashflow_record/PDB_cashflow_summary which use `isin_record_id`).

## 3. MANDATORY QUERY PATTERNS

These patterns are extracted from production-validated queries. Follow them exactly.

### 3A. Maturity queries — always use latest_redemption CTE + perpetual handling

```sql
WITH latest_redemption AS (
    SELECT isin_id, MAX(redemption_date) AS redemption_date
    FROM public."PDB_redemption"
    GROUP BY isin_id
)
SELECT
    ir.isin,
    CASE
        WHEN ir.seniority = 'Perpetual' THEN ir.call_option_date
        ELSE lr.redemption_date
    END AS maturity_date
FROM public."PDB_isin_records" ir
JOIN public."PDB_issuer_organization" io ON ir.issuer_organization_id = io.id
LEFT JOIN latest_redemption lr ON ir.id = lr.isin_id
WHERE ir.suspended = false
    AND {issuer_filter}
    AND (
        CASE
            WHEN ir.seniority = 'Perpetual' THEN ir.call_option_date
            ELSE lr.redemption_date
        END
    ) BETWEEN CURRENT_DATE AND CURRENT_DATE + INTERVAL '{period}'
ORDER BY maturity_date;
```

**Rules:**
- Multiple redemption dates per ISIN → always use `MAX(redemption_date)` in CTE.
- Perpetual ISINs → use `ir.call_option_date` instead of redemption_date.
- For queries needing next call date from the multi-date table: use `MIN(call_option_date)` from PDB_call_option_dates.

### 3B. Issue date queries — always use first_payin CTE

```sql
WITH first_payin AS (
    SELECT isin_id, MIN(payin_date) AS issue_date
    FROM public."PDB_payin"
    GROUP BY isin_id
)
SELECT ir.isin, io.issuer_name, fp.issue_date
FROM public."PDB_isin_records" ir
JOIN public."PDB_issuer_organization" io ON ir.issuer_organization_id = io.id
JOIN first_payin fp ON ir.id = fp.isin_id
WHERE ir.suspended = false
    AND {issuer_filter}
    AND fp.issue_date >= CURRENT_DATE - INTERVAL '{period}'
ORDER BY fp.issue_date DESC;
```

**Rules:**
- Multiple payin dates per ISIN → always use `MIN(payin_date)` as issue date.

### 3C. Latest trade per ISIN — use DISTINCT ON

```sql
WITH latest_trade AS (
    SELECT DISTINCT ON (t.isin_record_id)
        t.isin_record_id,
        t.trade_date,
        t.trade_time,
        t.last_traded_price,
        t.last_traded_yield_percent
    FROM public."SDB_trade" t
    WHERE t.last_traded_yield_percent BETWEEN 0 AND 100
    ORDER BY t.isin_record_id, t.trade_date DESC, t.trade_time DESC
)
```

**Rules:**
- Always filter yields: `BETWEEN 0 AND 100` to exclude garbage data.
- Order by: `isin_record_id, trade_date DESC, trade_time DESC`.

### 3D. Weighted average yield/price (VWAP/VWAY) — manual calculation

```sql
SUM(t.last_traded_yield_percent * t.traded_value_rs) / NULLIF(SUM(t.traded_value_rs), 0) AS way
SUM(t.last_traded_price * t.traded_value_rs) / NULLIF(SUM(t.traded_value_rs), 0) AS wap
```

**Rules:**
- Always use NULLIF to prevent division by zero.
- Always filter `t.last_traded_yield_percent BETWEEN 0 AND 100`.
- GROUP BY `ir.id, io.issuer_name` (or as appropriate).

### 3E. Liquidity queries — use SDB_fifteen_days_trade_avg

```sql
JOIN public."SDB_fifteen_days_trade_avg" f ON f.isin = ir.isin
WHERE f.agg_vol > 0
    AND f."WAY" IS NOT NULL
    AND f.avg_daily_vol IS NOT NULL
```

**Rules:**
- Liquid = `agg_vol > 0` in the 15-day average table.
- Join via `f.isin = ir.isin` (text match, not ID).
- Sort by `f.avg_daily_vol DESC` or `f.agg_vol DESC` as appropriate.

### 3F. Remaining maturity window (tenor bucket)

```sql
AND (
    (
        CASE
            WHEN ir.seniority = 'Perpetual' THEN ir.call_option_date
            ELSE lr.redemption_date
        END
    ) - CURRENT_DATE
) / 365.0 BETWEEN {min_years} AND {max_years}
```

Or for "less than X years":
```sql
AND (... - CURRENT_DATE) / 365.0 < {max_years}
```

### 3G. Cashflow queries — join via isin_record_id

```sql
FROM public."PDB_cashflow_record" c
JOIN public."PDB_isin_records" ir ON c.isin_record_id = ir.id
WHERE ir.isin = '{ISIN_CODE}'
    AND c.cash_flow_date >= CURRENT_DATE  -- for future cashflows only
ORDER BY c.cash_flow_date;
```

- Total cashflow: `COALESCE(c.coupon_cash_flow, 0) + COALESCE(c.principal_cash_flow, 0) AS total_cash_flow`
- Next IP: add `AND c.coupon_cash_flow IS NOT NULL ORDER BY c.cash_flow_date LIMIT 1`

### 3H. Tag-based filtering — use EXISTS or direct JOIN

For exact tag match (preferred):
```sql
AND EXISTS (
    SELECT 1 FROM public."PDB_tag" pt
    WHERE pt.isin_id = ir.id AND pt.tag = 'MLD'
)
```

Or for broader matching:
```sql
AND EXISTS (
    SELECT 1 FROM public."PDB_tag" pt
    WHERE pt.isin_id = ir.id AND LOWER(pt.tag) LIKE '%strip%'
)
```

**Known exact tag values:** 'PARTIAL REDEMPTION', 'PARTLY PAID', 'MLD', 'PERPETUAL', 'STRPP', 'FRB', 'PSU', 'TAXFREE', 'Zero Coupon', 'Green Bond', 'AT1', 'Tier 2', 'Subdebt', 'Plain Vanilla'

### 3I. Covenant queries — check NOT NULL and non-empty

```sql
WHERE ir.suspended = false
    AND ir.financial_covenants_min_nw IS NOT NULL
    AND TRIM(ir.financial_covenants_min_nw) <> ''
```

| User says | SQL filter |
|---|---|
| PSU, PSU bonds, PSU sector | `io.ownership = 'PSU'` |
| HFC, housing finance | `io.issuer_industry = 'HFC'` |
| NBFC | `io.issuer_industry = 'NBFC'` |
| Banks | `io.issuer_industry = 'Banks'` or `'Bank'` |

| Multiple issuers | `io.issuer_alias IN ('PFCLTD','IRFCLTD','RECLTD')` or use OR with prefix LIKE |
| "comparable to Bajaj Housing Finance" | `io.issuer_industry = 'HFC'` (same sector) |

**NEVER:**
- Join on `issuer_name` text match for filtering — always use `issuer_organization_id`.
- Use `issuer_industry` when user says "PSU" — use `ownership = 'PSU'` instead.
- Use `%rec%` style matching — use `rec%` prefix or exact IN match.

## 7. TABLE SELECTION RULES

**Do NOT use PDB_ebp_records when:**
- Querying face_value → use `ir.face_value`
- Querying coupon data → use `ir.coupon_fixed`, `ir.coupon_floating`
- Querying seniority, secured/unsecured → use `ir.seniority`, `ir.secured_or_unsecured`
- Querying tags → use PDB_tag
- Querying cashflows → use PDB_cashflow_record joined via `isin_record_id`
- Querying ratings → use PDB_current_rating_agency
- Querying record_date → use `ir.record_date`

**Use PDB_ebp_records only for:**
- `yield_ebp`, `spread_bps` (issuance), `total_issue_size`, `bidding_date`, `date_of_allotment`
- `Cover_Ratio`, `cut_off_yield`, `weighted_avg_cutoff_yield`
- Listing exchange (when ir.listing_exchange is null)
- Anchor investor data, QIB/non-QIB data

**For secondary market yield/price:** Use SDB_trade or SDB_fifteen_days_trade_avg, NEVER PDB_ebp_records.

## 12. SHUT PERIOD DETECTION PATTERN

```sql
WITH next_ip AS (
    SELECT
        c.isin_record_id,
        c.cash_flow_date,
        ROW_NUMBER() OVER (PARTITION BY c.isin_record_id ORDER BY c.cash_flow_date) AS rn
    FROM public."PDB_cashflow_record" c
    WHERE c.cash_flow_date >= CURRENT_DATE AND c.coupon_cash_flow IS NOT NULL
)
SELECT
    i.isin,
    n.cash_flow_date AS next_ip_date,
    CASE
        WHEN i.record_date::TEXT ~ '^\\d{8}$'
            AND CURRENT_DATE >= TO_DATE(i.record_date::TEXT, 'YYYYMMDD')
            AND CURRENT_DATE < n.cash_flow_date
        THEN 'YES' ELSE 'NO'
    END AS is_under_shut_period,
    (n.cash_flow_date - CURRENT_DATE) AS days_to_next_ip
FROM public."PDB_isin_records" i
JOIN next_ip n ON i.id = n.isin_record_id AND n.rn = 1
WHERE i.isin = '{ISIN_CODE}';
```

## 13. EDGE CASES & GUARDRAILS

1. **Perpetual bonds:** Always check `ir.seniority = 'Perpetual'` and substitute `ir.call_option_date` for maturity_date. Also detectable via `EXTRACT(YEAR FROM lr.redemption_date) = 9999`.
2. **Rating variants:** AAA includes 'AAA (SO)' and 'AAA (CE)'. Always include all three.
3. **Alias matching:** Use prefix match (`LIKE 'pfc%'`) or exact IN list. NEVER use `%pfc%` or `ILIKE '%REC%'` (catches PRECTF, DIRECTV, etc.).
4. **Suspended ISINs:** Every query with ir must include `ir.suspended = false`.
5. **Yield sanity:** Always filter `last_traded_yield_percent BETWEEN 0 AND 100` on SDB_trade.
6. **Covenant text fields:** Check both `IS NOT NULL` and `TRIM(...) <> ''`.
7. **GOI serviced / unsecured:** These are stored in `ir.seniority`, not in `PDB_isin_security` or `ir.secured_or_unsecured`.
8. **Zero coupon:** Filter via `ir.coupon_fixed = 0.0000`, not via tag.
9. **Floating rate:** Filter via `ir.coupon_floating IS NOT NULL`, not via tag.
10. **ISIN-specific queries:** When user provides an ISIN code, filter directly: `ir.isin = '{CODE}'`. Do NOT join PDB_ebp_records unless you need EBP-specific fields.
11. **Cashflows:** Join via `c.isin_record_id = ir.id`, NOT via PDB_ebp_records.
12. **WAY/WAP columns:** Always double-quote: `f."WAY"`, `f."WAP"`.

## 14. PRAFUL'S STYLE & FORMATTING CONVENTIONS

The output query must look and read like Praful wrote it. These conventions are extracted from the production query corpus.

### 14A. Schema prefix
- **Default to `public."TableName"` for every table reference.** Praful uses the explicit `public.` schema prefix in roughly 75% of queries. When in doubt, use it. The bare `"TableName"` form is acceptable for short queries (single-table SELECT, basic 2-table joins) but `public.` is the safer default.
- Both forms must always double-quote the table name because all tables use mixed case (PDB_, SDB_).

### 14B. Indentation
- 4 spaces per indent level. Never tabs.
- Each column in SELECT goes on its own line, indented 4 spaces from the SELECT keyword.
- Each JOIN goes on its own line at the FROM-level indent.
- Each ON clause is indented 4 spaces below its JOIN line — Praful frequently writes joins as:
  ```sql
  JOIN public."PDB_issuer_organization" io
      ON ir.issuer_organization_id = io.id
  ```
- WHERE clauses: each AND condition starts on a new line, aligned with WHERE, indented further when wrapped.

### 14C. Capitalization
- SQL keywords UPPERCASE: `SELECT`, `FROM`, `JOIN`, `LEFT JOIN`, `WHERE`, `AND`, `OR`, `IN`, `IS NOT NULL`, `BETWEEN`, `CASE`, `WHEN`, `THEN`, `ELSE`, `END`, `AS`, `ORDER BY`, `GROUP BY`, `LIMIT`, `WITH`, `EXISTS`, `DISTINCT`, `INTERVAL`, `CURRENT_DATE`.
- Aliases lowercase: `ir`, `io`, `cra`, `e`, `t`, `f`, `lr`, `fp`, `ncd`, `c`, `pt`, `sec`, `lt`.
- Column names lowercase as they exist in DB. Quoted uppercase columns ONLY when DB stores them uppercase: `f."WAY"`, `f."WAP"`, `t."ISIN"`.

### 14D. CTE naming convention
Praful's CTEs follow predictable names — use these exactly:
- `latest_redemption` — for `MAX(redemption_date) GROUP BY isin_id`
- `first_payin` — for `MIN(payin_date) GROUP BY isin_id`
- `latest_trade` — for `DISTINCT ON (isin_record_id)` from SDB_trade
- `next_call_date` — for `MIN(call_option_date) GROUP BY isin_id` from PDB_call_option_dates
- `next_ip` — for the next interest payment date with ROW_NUMBER pattern
- `final_data` — outer CTE wrapping computed maturity_date when extra filtering is needed afterward

### 14E. Output column aliasing
- `ir.isin` is normally output as `ir.isin` (no alias). For Q3 and a few others, Praful uses `ir.isin AS isin` — this is acceptable but unnecessary unless disambiguating.
- Maturity computed via CASE → always alias as `maturity_date`.
- Issue date from `MIN(payin_date)` → always alias as `issue_date`.
- Total cashflow → alias as `total_cash_flow`.
- VWAP → alias as `wap` (lowercase). VWAY → alias as `way` (lowercase).
- EBP yield → alias as `issuance_yield`.
- Next IP date → alias as `next_ip_date`.
- For shut period: `is_under_shut_period`, `days_to_next_ip`, `shut_period_days`.

### 14F. ORDER BY conventions
- Maturity queries → `ORDER BY maturity_date` (ascending — earliest maturing first).
- Issue date / payin queries → `ORDER BY fp.issue_date DESC` (most recent first).
- Trade volume queries → `ORDER BY f.avg_daily_vol DESC` or `f.agg_vol DESC`.
- Yield rankings → `ORDER BY {yield} DESC` (highest first).
- Listing queries (general) → `ORDER BY io.issuer_name, ir.isin` (alphabetical).
- Record date queries → `ORDER BY ir.record_date`.
- Past call/put dates → `ORDER BY ir.call_option_date DESC` (most recent past first).
- Use `NULLS LAST` when the sort column may have nulls and you want them at the bottom: `ORDER BY f.agg_vol DESC NULLS LAST`.

### 14G. WHERE clause ordering
Praful orders WHERE conditions in this priority sequence:
1. `ir.suspended = false` (ALWAYS first or near-first)
2. Issuer filters (`io.ownership = 'PSU'` or `LOWER(io.issuer_alias) LIKE 'pfc%'`)
3. Rating filters (`cra.rating IN ('AAA','AAA (SO)','AAA (CE)')`)
4. Attribute filters (face_value, coupon_fixed, listing_exchange, etc.)
5. Trade-data sanity filters (`yield BETWEEN 0 AND 100`, `agg_vol > 0`)
6. Date/range filters (`BETWEEN CURRENT_DATE AND ...`, `>= CURRENT_DATE - INTERVAL ...`)

This order is not strict but matches his typical structure.

## 16. CRITICAL ANTI-PATTERNS (NEVER DO THIS)

These are mistakes Praful explicitly flagged must consider :

1. ** Joining via issuer_name text match.** ALWAYS use `ir.issuer_organization_id = io.id`. Never `e.issuer_name = io.issuer_name`.

2. ** Using PDB_ebp_records for ISIN-level attributes.** EBP is for issuance-event data (yield_ebp, spread_bps, bidding_date, allotment). Anything that is a property of the bond itself (face value, coupon, seniority, listing, tags, cashflows, ratings, record_date) lives on PDB_isin_records or its child tables.

3. ** Wildcard alias matching `%REC%` or `%pfc%`.** This catches false positives (PRECTF, DIRECTV). Always use prefix match `'rec%'` or exact `IN ('RECLTD',...)`.

4. ** Using `issuer_industry = 'PSU'`.** PSU is an ownership classification. Use `io.ownership = 'PSU'`. The `issuer_industry` field has values like 'HFC', 'NBFC', 'Banks', 'INFRA', 'Manufacturing' — `'PSU'` exists there too but is unreliable for PSU detection.

5. ** Single-rating filter `rating = 'AAA'`.** AAA bonds also exist as 'AAA (SO)' (structured obligation) and 'AAA (CE)' (credit enhancement). Always: `cra.rating IN ('AAA', 'AAA (SO)', 'AAA (CE)')`. Apply same logic to AA+, AA, etc., if SO/CE variants are likely.

6. ** Using EBP yield (`e.yield_ebp`) as market yield.** EBP yield is the issuance yield (one-time, at allotment). It is NOT reliable for current market levels. For market levels, use SDB_trade (latest) or SDB_fifteen_days_trade_avg.

7. ** Using ILIKE '%partial%' for tag matching.** The exact tag is `'PARTIAL REDEMPTION'` (uppercase, with space). Use `UPPER(t.tag) = 'PARTIAL REDEMPTION'` or exact match `pt.tag = 'PARTIAL REDEMPTION'`.

8. ** Forgetting `ir.suspended = false`.** Every query touching PDB_isin_records MUST include this filter to exclude inactive ISINs. No exceptions.

9. ** Forgetting yield sanity bounds.** SDB_trade has dirty data — yields can be 0, negative, or above 1000. Always: `t.last_traded_yield_percent BETWEEN 0 AND 100`.

10. ** Joining cashflows through PDB_ebp_records.** Cashflows are ISIN-level. Join `PDB_cashflow_record c` directly to `PDB_isin_records ir` via `c.isin_record_id = ir.id` (note the column name: `isin_record_id`, not `isin_id`).

11. ** Lowercase unquoted WAY/WAP.** Postgres folds unquoted identifiers to lowercase. The actual column is uppercase `"WAY"`. Writing `f.way` will produce a "column does not exist" error. Always: `f."WAY"`, `f."WAP"`. Same for `t."ISIN"` in SDB_trade.

12. ** Wrong join column on PDB_current_rating_agency.** The column is `cra.isin_id`, NOT `cra.isin_record_id`. (The `isin_record_id` naming convention is used by `PDB_cashflow_record` and `SDB_trade`; do not generalize.)

13. ** Using `tenor_nse_months` or similar EBP fields when redemption logic is needed.** Always derive tenor from redemption_date using the perpetual CASE pattern.

14. ** Joining PDB_redemption directly without aggregation.** Multiple redemption rows per ISIN (amortizing/staggered) will multiply rows. Always wrap in a CTE with `MAX(redemption_date) GROUP BY isin_id`.

15. ** Joining PDB_payin directly without aggregation.** Same issue — partly-paid bonds have multiple payin rows. Always use `MIN(payin_date) GROUP BY isin_id`.

## 17. NUANCED RULES (FROM PRAFUL'S COMMENTS, VERBATIM REASONING)

### 17A. On maturity logic (Q1, Q4.1, Q5.1, Q5.2)
> "For ISINs with multiple redemption dates, the latest (MAX) redemption_date per ISIN should be considered. For perpetual ISINs, redemption_date is not relevant — instead call_option_date should be used from call option data."

This is non-negotiable. The CASE expression:
```sql
CASE WHEN ir.seniority = 'Perpetual' THEN ir.call_option_date ELSE lr.redemption_date END
```
must be repeated in WHERE, SELECT, and ORDER BY when used (Postgres won't let you reference a SELECT alias in WHERE).

### 17B. On PSU filtering (Q1, Q4.1, Q19, Q41, Q42)
> "Avoid using issuer_industry and joins on issuer_name; use ownership = 'PSU' with ID-based joins."

`ownership = 'PSU'` is the canonical PSU filter. It includes both Central PSUs and (depending on data) state PSUs. If user specifies "Central PSU" specifically, use `ownership = 'PSU'`; if "State Government", use `ownership = 'State Government'`.

### 17C. On alias matching (Q7, Q8, Q43)
> "Alias matching should use prefix (rec%) instead of %rec% to prevent incorrect matches. Use of ILIKE '%REC%' can lead to incorrect matches (e.g., PRECTF)."

When the user names specific issuers, prefer the exact alias from the known list:
- PFC → `'PFCLTD'`
- IRFC → `'IRFCLTD'`
- REC → `'RECLTD'`
- NABARD → `'NABARD'`
- NHPC → `'NHPC'`
- HUDCO → `'HUDCO'`
- NHAI → `'NHAI'`

For multi-issuer filtering, use either:
```sql
io.issuer_alias IN ('IRFCLTD', 'PFCLTD', 'RECLTD')
```
OR (when prefix matching is acceptable for partial entries):
```sql
LOWER(io.issuer_alias) LIKE 'pfc%'
OR LOWER(io.issuer_alias) LIKE 'rec%'
OR LOWER(io.issuer_alias) LIKE 'irfc%'
```
Praful uses the IN form for clean canonical issuers and the LIKE form when there might be tranches/sub-aliases (PFCLTD-1, etc.).

### 17D. On listing detection (Q13)
> "'Listed bonds' can be identified using listing_exchange IS NOT NULL, but join should be via IDs, not indirect usage."

Two valid forms:
- `ir.listing_exchange IS NOT NULL` (preferred — no extra join)
- `e.listing_exchange IS NOT NULL` (acceptable when EBP is already joined for other reasons)

Do not double-filter on both unless deliberately checking consistency.

### 17E. On market yield source (Q13)
> "Yield should be taken from trade data (SDB_trade), not PDB_ebp_records, since EBP yield is not reliable for market levels."

For "yielding more than 10%" or any current market yield filter:
```sql
JOIN public."SDB_trade" t ON t.isin_record_id = ir.id
WHERE t.last_traded_yield_percent > 10
    AND t.last_traded_yield_percent BETWEEN 0 AND 100
```
For "issuance yield more than 10%":
```sql
WHERE e.yield_ebp > 10
```

### 17F. On VWAP/VWAY aggregation (Q41, Q42)
> "Apply yield filter (0–100) and ensure VWAP logic is used correctly for aggregation."

Canonical aggregation pattern for date-range volume-weighted yield:
```sql
SELECT
    ir.isin AS isin,
    io.issuer_name,
    SUM(t.last_traded_yield_percent * t.traded_value_rs)
        / NULLIF(SUM(t.traded_value_rs), 0) AS way
FROM public."SDB_trade" t
JOIN public."PDB_isin_records" ir ON t.isin_record_id = ir.id
JOIN public."PDB_issuer_organization" io ON ir.issuer_organization_id = io.id
JOIN public."PDB_current_rating_agency" cra ON ir.id = cra.isin_id
WHERE io.ownership = 'PSU'
    AND cra.rating IN ('AAA','AAA (SO)','AAA (CE)')
    AND t.last_traded_yield_percent BETWEEN 0 AND 100
    AND t.trade_date BETWEEN DATE '{start}' AND DATE '{end}'
GROUP BY ir.id, io.issuer_name;
```
Note: `ir.suspended = false` was omitted in Q41/Q42 of the corpus, but include it for safety.

### 17G. On YTM / YTC / YTP (Q36, Q37, Q38)
> "YTM is calculated when: yield_to_price = True, ytm is provided, selected_yield = 'ytm' with clean price. Maturity date must be after value date."

These yields cannot be computed in pure SQL. The query should return the raw reference data for downstream calculation:
- For YTM: `did, cashflow_link, face_value, record_date, redemption_date AS maturity_date`
- For YTC: `did, cashflow_link, face_value, call_option_date, record_date`
- For YTP: `did, cashflow_link, face_value, put_option_date, record_date`

Filter conditions:
- YTC: `ir.call_option_date IS NOT NULL`
- YTP: `ir.put_option_date IS NOT NULL`
- YTM: redemption row from PDB_redemption (use `r.file_name_tag = 'PDB File'` filter and `LIMIT 1` ordered by date)

### 17H. On rating queries by ISIN (Q14)
When asked for "the rating of ISIN X" and an ISIN can have multiple agency ratings, use STRING_AGG:
```sql
SELECT
    ir.isin,
    STRING_AGG(cra.rating_agency, ', ') AS rating_agencies,
    STRING_AGG(cra.rating, ', ') AS ratings
FROM public."PDB_current_rating_agency" cra
JOIN public."PDB_isin_records" ir ON cra.isin_id = ir.id
WHERE ir.isin = 'INE403Q07GO9'
GROUP BY ir.isin;
```
Mention NSDL as the source if the user asks where ratings come from.

### 17I. On EXISTS vs JOIN for tag filtering (Q20, Q23, Q24)
Use `EXISTS` when the user asks for a tag-based property and you don't need to display the tag itself:
```sql
WHERE EXISTS (
    SELECT 1 FROM public."PDB_tag" pt
    WHERE pt.isin_id = ir.id AND pt.tag = 'MLD'
)
```
Use direct JOIN when the user wants to see the tag value or when filtering tag combined with other tag attributes:
```sql
JOIN public."PDB_tag" t ON e.isin_id = t.isin_id
WHERE UPPER(t.tag) = 'PARTLY PAID'
```
EXISTS is the preferred default — it avoids row multiplication when an ISIN has multiple tags.

### 17J. On future-only cashflows (Q49.1, Q49.2)
For "next IP" or "remaining cashflows": filter `c.cash_flow_date >= CURRENT_DATE`. For "next IP", additionally filter `c.coupon_cash_flow IS NOT NULL` (excludes pure principal-only cashflows) and `LIMIT 1 ORDER BY c.cash_flow_date`.

### 17K. On shut period (Q12)
The record_date column is an integer (days before IP), but in production data some rows store it as a YYYYMMDD-format integer. The shut-period query handles this via:
```sql
CASE WHEN i.record_date::TEXT ~ '^\\d{8}$'
     THEN TO_DATE(i.record_date::TEXT, 'YYYYMMDD')
END
```
Use this exact pattern when computing shut period. Bond is in shut period when `CURRENT_DATE >= record_date AND CURRENT_DATE < next_ip_date`.

| User says | SQL |
|---|---|
| "next 30 days" / "next month" | `BETWEEN CURRENT_DATE AND CURRENT_DATE + INTERVAL '30 DAY'` |
| "next quarter" / "3 months" | `INTERVAL '3 MONTH'` |
| "next 6 months" / "six months" / "half year" | `INTERVAL '6 MONTH'` |
| "next 1 year" / "next year" | `INTERVAL '1 YEAR'` |
| "next 2 years" | `INTERVAL '2 YEAR'` |
| "last week" | `>= CURRENT_DATE - INTERVAL '7 DAY'` |
| "last month" / "30 days" | `>= CURRENT_DATE - INTERVAL '1 MONTH'` |
| "last 3 months" / "last quarter" | `>= CURRENT_DATE - INTERVAL '3 MONTH'` |
| "last 6 months" | `>= CURRENT_DATE - INTERVAL '6 MONTH'` |
| "last year" / "last 1 year" | `>= CURRENT_DATE - INTERVAL '1 YEAR'` |
| "in 2024" / "in FY24" | `EXTRACT(YEAR FROM date_col) = 2024` (calendar) — for FY use `BETWEEN DATE '2023-04-01' AND DATE '2024-03-31'` (Indian FY) |
| "between Jan 2025 and Mar 2026" | `BETWEEN DATE '2025-01-01' AND DATE '2026-03-31'` |
| "between x date and y date" | `BETWEEN DATE '{x}' AND DATE '{y}'` |
| "in last 15 days" | use `SDB_fifteen_days_trade_avg` (precomputed) — do NOT compute manually unless required |
| "remaining maturity 2.5–3.5Y" | `((maturity_expr) - CURRENT_DATE) / 365.0 BETWEEN 2.5 AND 3.5` |

**Indian Financial Year:** April 1 to March 31. FY24 = Apr 2023 to Mar 2024. FY26 = Apr 2025 to Mar 2026. Always interpret "FY" as Indian fiscal year unless user says "calendar year".

**Indian numerical conventions:**
- 1 lakh = 100,000 (1L)
- 1 crore = 10,000,000 (1Cr)
- 1 thousand = 1,000 (1K)
- "100 cr" = 1,000,000,000 (₹100 crore = ₹1 billion)
- Issue size in DB is already in Crores (e.g., `total_issue_size = 500` means ₹500 Cr)
- Face value in DB is in absolute Rupees (`face_value = 1000000` means ₹10 lakh per bond)
- Trade value `traded_value_rs` is in absolute Rupees (not Cr)

## 19. CANONICAL QUERY SHAPES BY INTENT CATEGORY

### Category A: "All [issuer-filter] [property-filter]"
**Shape:** SELECT from PDB_isin_records → JOIN issuer → optional CTEs for redemption/payin → WHERE clauses.
**Example template:**
```sql
SELECT ir.isin, io.issuer_name, {extra_columns}
FROM public."PDB_isin_records" ir
JOIN public."PDB_issuer_organization" io ON ir.issuer_organization_id = io.id
{optional_joins}
WHERE ir.suspended = false
    AND {issuer_filter}
    AND {property_filters}
ORDER BY {natural_sort};
```

### Category B: "[Issuer] maturing in [period]"
Use 3A pattern (latest_redemption CTE + perpetual CASE).

### Category C: "[Issuer] issued in [period]"
Use 3B pattern (first_payin CTE).

### Category D: "Liquid [issuer] securities" / "Top [N] traded"
Use SDB_fifteen_days_trade_avg with `f.agg_vol > 0`. Sort by `f.avg_daily_vol DESC` or `f.agg_vol DESC`.

### Category E: "[Securities] traded between [date X] and [date Y]"
Use SDB_trade aggregation with VWAY/VWAP formula. GROUP BY ir.id, io.issuer_name.

### Category F: "Latest traded levels of [securities]"
Use latest_trade CTE with `DISTINCT ON (isin_record_id)`. Optionally JOIN with SDB_fifteen_days_trade_avg for 15-day context.

### Category G: "Specific ISIN lookup" (record date, rating, cashflow, IP)
Direct filter `WHERE ir.isin = '{CODE}'`. Single-table or 2-table query. No EBP join unless asking for issuance data.

### Category H: "Comparable to [issuer]"
Identify the sector via the named issuer's industry, then find peers in the same `issuer_industry` with similar rating. Optionally include the named issuer for direct comparison.

### Category I: "Covenant: [covenant_type]"
Single-table query on PDB_isin_records. Filter `IS NOT NULL AND TRIM(...) <> ''`. Sort by `io.issuer_name, ir.isin`.

### Category J: "Cashflow / IP / shut period for ISIN"
PDB_cashflow_record joined via `c.isin_record_id = ir.id`. Filter `c.cash_flow_date >= CURRENT_DATE` for future cashflows. Use ROW_NUMBER for shut period.

## 20. WORKED PATTERN: SECTOR + RATING + LIQUIDITY + MATURITY (Q5.2)

This is the most complex single-query pattern in the corpus. Memorize this skeleton — most multi-condition questions follow it:

```sql
WITH latest_redemption AS (
    SELECT
        isin_id,
        MAX(redemption_date) AS redemption_date
    FROM public."PDB_redemption"
    GROUP BY isin_id
)
SELECT
    ir.isin,
    io.issuer_name,
    cra.rating,
    CASE
        WHEN ir.seniority = 'Perpetual'
        THEN ir.call_option_date
        ELSE lr.redemption_date
    END AS maturity_date,
    f."WAY",
    f."WAP",
    f.avg_daily_vol,
    f.avg_vol_trades,
    f.agg_vol
FROM public."PDB_isin_records" ir
JOIN public."PDB_issuer_organization" io
    ON ir.issuer_organization_id = io.id
JOIN public."PDB_current_rating_agency" cra
    ON cra.isin_id = ir.id
JOIN public."SDB_fifteen_days_trade_avg" f
    ON f.isin = ir.isin
LEFT JOIN latest_redemption lr
    ON lr.isin_id = ir.id
WHERE
    ir.suspended = false
    AND io.ownership = 'PSU'
    AND cra.rating IN ('AAA','AAA (SO)','AAA (CE)')
    AND f."WAY" IS NOT NULL
    AND f.avg_daily_vol IS NOT NULL
    AND f.agg_vol > 0
    AND (
        (
            CASE
                WHEN ir.seniority = 'Perpetual'
                THEN ir.call_option_date
                ELSE lr.redemption_date
            END
        ) - CURRENT_DATE
    ) / 365.0 < 5
ORDER BY f.avg_daily_vol DESC;
```

Key lessons:
- LEFT JOIN for redemption (some bonds may have no redemption row, e.g., perpetuals).
- INNER JOIN for trade/liquidity (we want only liquid bonds, no nulls).
- The maturity CASE is repeated 3 times (SELECT, WHERE, implicitly in ORDER BY if used).
- All quoted uppercase columns: `f."WAY"`, `f."WAP"`.
- Liquidity sanity: `f."WAY" IS NOT NULL AND f.avg_daily_vol IS NOT NULL AND f.agg_vol > 0`.
